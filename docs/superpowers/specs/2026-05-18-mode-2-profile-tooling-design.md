# Mode-2 profile tooling (slice 3, host side) — design

Status: design approved 2026-05-18.

Parent design: `docs/superpowers/specs/2026-05-17-mode-2-design.md` — this is the
host side of that design's slice 3 (§5/§8). The mode-2 EFI runtime (slices 1–2)
shipped on branch `mode-2` / PR #28.

## 1. Goal & scope

Produce the host-side tooling that turns a stock `vbmeta.img` into a mode-2
profile and a staged-bootable test EFI:

```
vbmeta.img → gbl-chainload_profile.xml → mode2_profile.bin (120 B)
           → GBLP1 0x0010 overlay → dist/mode-2-test.efi
```

The end goal is one command — `bash scripts/make-mode2-test-overlay.sh` — that
emits `dist/mode-2-test.efi`, which the user stages on device with
`fastboot stage dist/mode-2-test.efi && fastboot oem boot-efi`. Under a
`--verbose` build the logs then show the mode-2 rewrites firing
(`mode2 | km-rewrite | cmd=0x00000201` etc.).

In scope:

- The vbmeta→XML helper.
- The `gbl-chainload_profile.xml` schema.
- The XML→`0x0010`-binary compiler.
- A `gbl-pack` flag to embed a profile entry.
- An orchestration script that emits `dist/mode-2-test.efi`.

Out of scope (remain follow-ups):

- The device-side aarch64 helper binary.
- The `images/`-drop full build orchestration.
- The mode-2 ZIP. (A device/ZIP-side profile parser is a later concern; the
  host tooling here is Python.)

### Key approach decision

The XML→binary compile step is **Python**, not C. `gbl-pack` is a C tool;
embedding an XML parser in C is fragile. `gbl-pack` therefore gains only a
`--mode2-profile <bin>` flag that embeds an already-built 120-byte binary as a
`0x0010` entry. All XML handling lives in the Python layer.

Rejected alternatives: `gbl-pack` parsing XML directly (C + XML); dropping the
XML authoring form (the chosen scope keeps it for human review/override).

## 2. Components

### `tools/mode2-profile/` — Python CLI

Host tool. The device-side aarch64 build is a later follow-up. One CLI
(`mode2-profile`) with two subcommands:

- **`derive <vbmeta.img> -o <profile.xml>`** — the vbmeta→XML helper. Reuses
  the proven logic of the canoe reference `tools/ota_to_overrides.py`:
  - Import `avbtool.py` via `$AVBTOOL` or `~/avbtool.py` (canoe's
    `_import_avbtool` pattern).
  - Pull the AVB public-key blob from the vbmeta aux block; compute
    `rot_digest = SHA256(pubkey ‖ 0x00)` and `pubkey_digest = SHA256(pubkey)`.
  - Parse the OS-version / security-patch property descriptors and encode them
    to the bootloader domain (`os_version = (Major<<14)|(Minor<<7)|SubMinor`;
    `spl = (Day<<11)|((Year-2000)<<4)|Month`).
  - Compute `vbh = sha256(header + authentication block + auxiliary block)` of
    the root vbmeta image, inline (chain-partition images are not required;
    this value differs from `avbtool calculate_vbmeta_digest` on images that
    carry chain descriptors).
  - Write the XML with `is_unlocked=0`, `color=0`.
- **`compile <profile.xml> -o <mode2_profile.bin>`** — the XML→binary compiler.
  Reads the XML and packs the 120-byte `gbl_mode2_profile` struct defined in
  `tools/shared/gbl_mode2_profile.h` (`GM2P` magic, version 1, reserved 0, the
  four scalars, the three 32-byte digests). Pure struct packing.

### `gbl-pack` extension

A `--mode2-profile <mode2_profile.bin>` flag embeds the 120-byte binary as a
GBLP1 `0x0010` entry. To support a profile-only test overlay, `gbl-pack` is
relaxed: when `--cached-abl` is omitted but `--mode2-profile` is given, it emits
a valid container holding just the `0x0010` entry (today it hard-requires a
`cached_abl`). A production overlay later passes both flags.

### `scripts/make-mode2-test-overlay.sh`

Orchestration. Builds the mode-2 EFI (`./scripts/build.sh --mode 2 --debug
--verbose`), runs `derive` → `compile` → `gbl-pack --mode2-profile`,
concatenates the `0x0010`-only container onto `dist/mode-2.efi`, and emits
`dist/mode-2-test.efi`. Takes the vbmeta path as an argument, defaulting to
`images/vbmeta-infiniti-IN-16.0.7.201.img`.

## 3. XML schema

`gbl-chainload_profile.xml` — human-editable, every spoofed value visible, with
a provenance comment:

```xml
<!-- generated by mode2-profile derive
     source: images/vbmeta-infiniti-IN-16.0.7.201.img
     sha256: <hex>   os_version: '16.0.7' -> 0x...   spl: '2026-...' -> 0x... -->
<gbl-chainload-mode2-profile version="1">
  <is-unlocked>0</is-unlocked>
  <color>0</color>                          <!-- 0=GREEN -->
  <system-version>0x40000</system-version>
  <system-spl>0x9A4</system-spl>
  <rot-digest>44149b5d…c7</rot-digest>       <!-- 64 hex = SHA256(pubkey‖0x00) -->
  <pubkey-digest>8d897f62…bb</pubkey-digest> <!-- 64 hex = SHA256(pubkey) -->
  <vbh>…</vbh>                                <!-- 64 hex = sha256(root vbmeta header+auth+aux) -->
</gbl-chainload-mode2-profile>
```

The XML stores the derived digests as hex, not the raw ~1 KB public-key blob —
so a reviewer sees exactly what gets spoofed and `compile` is a straight
hex→bytes pack. Every field is hand-overridable. `compile` validates ranges
(`is_unlocked ≤ 1`, `color ≤ 3`, each digest exactly 64 hex characters,
`version` is 1).

## 4. Error handling & testing

Error handling:

- `derive`: clear errors for a missing or unsigned vbmeta, a missing
  `avbtool.py`, and missing OS-version / SPL property descriptors (port the
  canoe `ota_to_overrides.py` messages).
- `compile`: reject malformed XML, out-of-range fields, and wrong-length
  digests. Fail loud; never emit a bad binary.
- `gbl-pack`: a profile-only invocation must still produce a structurally valid
  GBLP1 container (header CRC, per-entry SHA, footer).

Testing — host tests under `tests/host/`:

- **Compiler↔parser agreement:** `compile` a known XML → feed the resulting
  binary to the existing `mode2_harness profile-parse` (the EDK2 parser from
  mode-2 Task 1) → expect `status=0`. This proves the Python compiler and the C
  parser agree byte-for-byte.
- **Real-fixture derive:** `derive` on `images/vbmeta-infiniti-IN-16.0.7.201.img`
  → the resulting XML `compile`s and the binary parses clean.
- **gbl-pack profile-only:** a `--mode2-profile`-only container is located by
  `parser_harness find-mode2-profile` with `status=0`.

The on-device step (`fastboot stage dist/mode-2-test.efi` +
`fastboot oem boot-efi`) stays manual / user-run — it is the CLAUDE.md-sanctioned
RAM-load test path. The deliverable is the `dist/mode-2-test.efi` artifact plus
the recipe.

## 5. Implementation decomposition (PR slices)

One design, landed as feature branches / PRs against `main`:

1. `tools/mode2-profile/` Python CLI — `derive` + `compile` subcommands +
   the XML schema + the compiler↔parser and real-fixture host tests.
2. `gbl-pack` `--mode2-profile` flag + profile-only-container relaxation +
   the gbl-pack profile-only host test.
3. `scripts/make-mode2-test-overlay.sh` orchestration.

Slice 3 depends on slices 1 and 2. The tooling depends on the mode-2 runtime
(`tools/shared/gbl_mode2_profile.h`, `mode2_harness`) from branch `mode-2` /
PR #28, so this work branches from `mode-2` until that PR merges.
